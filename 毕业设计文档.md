# 直接式汽车胎压状态监测系统的设计

## 摘要

本文设计了一套基于 STM32F103C8T6 微控制器的直接式汽车胎压监测系统（TPMS）。系统由轮胎端从机采集节点和仪表台端主机显示报警单元组成。从机通过 ADC 采集气压和温度传感器信号，经 SPI 总线发送至主机；主机接收数据后在 SSD1306 OLED 屏幕上实时显示，并在参数异常时驱动 LED 和蜂鸣器发出声光报警。系统采用 STM32CubeMX 配置外设并生成 HAL 库初始化代码，Keil MDK 编写应用固件，Proteus 平台完成仿真验证。仿真结果表明，系统能实时监测胎压（0~600 kPa）与温度（0~60°C），超限报警功能正常，各项指标满足设计要求。

**关键词**：TPMS；胎压监测；STM32F103C8T6；SPI 通信；Proteus 仿真

## 一、绪论

### 1.1 课题背景

汽车胎压监测系统（Tire Pressure Monitoring System, TPMS）是汽车安全领域的重要配置。直接式 TPMS 通过安装在轮胎内的传感器直接测量气压和温度，精度高、响应快，是当前的主流技术路线。本课题设计一套直接式 TPMS 仿真原型，以 STM32F103C8T6 为主控芯片，完成从传感器数据采集、数据传输到显示报警的完整功能链路。

### 1.2 主要研究内容

1. STM32F103C8T6 最小系统设计（时钟、复位、电源）
2. ADC 多通道采集——气压和温度传感器接口设计与数据获取
3. SPI 通信模块——从机发送传感器数据，主机接收并校验，作为 2.4GHz 无线通信的仿真替代
4. OLED 显示屏驱动——SSD1306 I2C 接口初始化与数据显示
5. 声光报警模块——LED 状态指示与蜂鸣器 PWM 驱动
6. 按键交互——显示模式切换与报警测试

## 二、系统总体方案

### 2.1 系统架构

系统由两个 STM32F103C8T6 微控制器组成，通过 5 线 SPI 直连实现数据交换：

- **从机（轮胎传感器节点）**：负责采集气压和温度数据，封装为数据帧后通过 SPI 发送至主机
- **主机（仪表台显示单元）**：负责接收从机数据帧、校验解析、OLED 显示、超限报警

**数据流**：从机每次主循环读取 ADC（气压 + 温度），构建 6 字节数据帧，等待主机拉低片选信号后通过 SPI 发送；主机以约 500ms 为周期拉低片选发起传输，接收数据帧、校验，校验通过后更新 OLED 显示并执行报警逻辑。

### 2.2 通信协议设计

#### 2.2.1 SPI 通信方案选择

SPI（Serial Peripheral Interface）是一种高速全双工同步串行通信协议，由四根信号线组成：SCK（时钟）、MOSI（主发从收）、MISO（从发主收）、SS（片选）。本系统选择 SPI 作为主从通信协议，基于以下考虑：

1. **全双工特性**：主机发送哑字节的同时即可接收从机数据，一次传输完成双向数据交换
2. **主从架构天然匹配**：SPI 的主从模式与本系统的集中式轮询架构完全对应
3. **时序可控**：时钟由主机产生，从机被动响应，通信节奏完全由主机掌握
4. **可扩展性**：实物系统中 SPI 可直接驱动 nRF24L01 等无线模块，仿真到实物的迁移成本低

#### 2.2.2 数据帧设计

从机采集气压（0~600 kPa）和温度（0~60°C），数据帧格式如下：

| 字节偏移 | 字段 | 内容 | 说明 |
|----------|------|------|------|
| Byte 0 | Sync | 0x00 | 同步字节，用于帧对齐 |
| Byte 1 | Header | 0x5A | 帧头标识 |
| Byte 2 | P_hi | 0x00~0x02 | 气压高字节 |
| Byte 3 | P_lo | 0x00~0xFF | 气压低字节 |
| Byte 4 | Temp | 0x00~0x3C | 温度值 |
| Byte 5 | Checksum | — | 累加和校验 |

**校验和计算**：Checksum = (Header + P_hi + P_lo + Temp) & 0xFF，主机收到帧后重新计算并与接收值比对，不一致则丢弃该帧。

**同步字节设计**：软件模拟 SPI 时，主从时序存在微小的不确定性。在最前端放置 0x00 同步字节作为保护间隔，主机从 Byte 1 开始解析有效数据，即便出现单字节的位偏移也不会影响帧头识别。

#### 2.2.3 软件 GPIO 模拟 SPI

本系统选择通过 GPIO 软件模拟 SPI 时序而非使用 STM32 硬件 SPI 外设，原因如下：

1. **跨平台一致性强**：软件 SPI 不依赖特定 MCU 外设型号，代码可在不同芯片间直接移植
2. **时序完全可控**：开发人员可精确控制 SCK 频率、数据建立/保持时间，便于满足不同速率外设的时序要求
3. **调试透明**：GPIO 电平可在仿真环境中直接观测，便于定位通信故障
4. **教学价值**：通过软件实现可深入理解 SPI 协议底层时序，符合毕业设计的教学目的

**SCK 频率**选择约为 500 Hz（半周期 1ms），低速通信可确保从机 GPIO 轮询有充足的响应时间窗，避免因 MCU 负载波动导致丢步。

**主机软件 SPI 时序**：拉低 SS→延时 50ms 等待从机就绪→逐字节发送，每 bit 依次拉高 SCK、延时 1ms、读取 MISO、拉低 SCK、延时 1ms→拉高 SS。

**从机软件 SPI 时序**：轮询 SCK 上升沿→在上升沿将当前 bit 输出到 MISO→移位准备下一 bit→轮询 SCK 下降沿→等待下一上升沿。

### 2.3 报警阈值设计

参照乘用车胎压推荐标准（GB/T 2978-2014，建议气压 200~280 kPa），设定以下阈值：

| 参数 | 正常范围 | 报警条件 |
|------|---------|---------|
| 气压 | 180 ~ 350 kPa | < 180 kPa（低压报警）/ > 350 kPa（高压报警） |
| 气压（严重） | — | < 150 kPa（严重低压，最高优先级） |
| 温度 | 0 ~ 45°C | < 0°C（低温报警）/ > 45°C（高温报警） |

报警优先级：严重低压 > 普通气压异常 > 温度异常。报警触发时红色 LED 亮、绿色 LED 灭、蜂鸣器发声。按下报警测试键可强制触发报警用于功能验证。

## 三、硬件设计

### 3.1 主控芯片

STM32F103C8T6，ARM Cortex-M3 内核，LQFP48 封装，主要特性：

| 参数 | 规格 |
|------|------|
| 最高主频 | 72 MHz |
| Flash / SRAM | 64 KB / 20 KB |
| ADC | 2×12-bit，最多 16 通道 |
| SPI | 2 路 |
| I2C | 2 路 |
| USART | 3 路 |
| 通用定时器 | 3 个（TIM2/3/4） |

仿真中系统时钟配置为 HSI 8MHz 内部 RC 振荡器。HSI 作为内部时钟源，无需外部晶振即可工作，上电启动快、不受仿真环境中晶体模型精度影响、适合原型验证。实物系统可替换为 HSE 8MHz 晶振 + PLL 倍频至 72MHz 以获得更高性能。

### 3.2 从机外设配置

从机安装于轮胎内，由电池供电，外设尽量精简。

| 外设 | 引脚 | 模式 | 说明 |
|------|------|------|------|
| 气压传感器 | PA1 | Analog | ADC1 通道 1，外接电位器分压模拟 |
| 温度传感器 | PA2 | Analog | ADC1 通道 2，外接电位器分压模拟 |
| SPI NSS | PA4 | Input | 接收主机片选信号 |
| SPI SCK | PA5 | Input（GPIO） | 接收主机时钟 |
| SPI MISO | PA6 | Output PP（GPIO） | 向主机发送数据 |
| SPI MOSI | PA7 | Input | 连接至主机 MOSI |
| 调试串口 | PA9 (TX) | AF PP | 输出传感器采样值 |
| 发送指示 | PB0 | Output PP | 每次 SPI 传输翻转 |
| 状态指示 | PC13 | Output PP | 系统运行指示 |

### 3.3 主机外设配置

主机安装于仪表台，外设丰富，实现显示和报警功能。

| 外设 | 引脚 | 模式 | 说明 |
|------|------|------|------|
| SPI SS | PA4 | Output PP | 软件控制片选 |
| SPI SCK | PA5 | Output PP（GPIO） | 软件产生时钟 |
| SPI MISO | PA6 | Input（GPIO） | 读取从机数据 |
| SPI MOSI | PA7 | Output PP（GPIO） | 发送哑字节 |
| OLED SCL | PB6 | Output OD | SSD1306 I2C 时钟线 |
| OLED SDA | PB7 | Output OD | SSD1306 I2C 数据线 |
| 蜂鸣器 | PA1 | AF PP / Analog | TIM2_CH2 PWM 输出 |
| 红色 LED | PB1 | Output PP | 报警指示 |
| 绿色 LED | PB0 | Output PP | 正常指示 |
| 蓝色 LED | PC13 | Output PP | 心跳指示 |
| 按键1 | PA0 | Input PU | 显示模式切换 |
| 按键2 | PA2 | Input PU | 报警测试 |
| 调试串口 | PA9 (TX) | AF PP | 输出接收数据与状态 |

### 3.4 OLED 显示模块

选用 SSD1306 驱动的 0.96 英寸 OLED，分辨率 128×64 像素，I2C 接口（地址 0x3C，写操作 0x78）。显示屏纵向分为 8 个页（Page），每页 128 列。采用 8×16 点阵 ASCII 字库，每字符占 16 列×2 页，全屏可显示 4 行×16 列英文字符。

### 3.5 蜂鸣器驱动电路

蜂鸣器采用 NPN 三极管（S8050）驱动方案。PA1 输出 PWM 信号经 1kΩ 限流电阻接入三极管基极，集电极接蜂鸣器至 VCC，发射极接地。基极并联 10kΩ 下拉电阻防止上电瞬间误触发。

**正常态噪声抑制**：蜂鸣器正常态需绝对静音。考虑到 SPI 通信时 GPIO 端口上多根引脚同时翻转可能通过芯片内部产生微弱耦合，本设计采取引脚模式切换策略：报警时将 PA1 设为 AF_PP（复用推挽输出），由 TIM2 输出 PWM 驱动蜂鸣器；正常时将 PA1 设为 Analog（模拟输入），数字输出电路彻底断开，消除任何潜在的数字噪声路径。

### 3.6 按键与 LED 电路

按键采用内部上拉方式，一端接 GPIO，另一端接 GND。常态 GPIO 读到高电平，按下时读到低电平。软件以 30ms 延迟做消抖处理。LED 串联 220Ω 限流电阻，由 GPIO 推挽输出直接驱动。

### 3.7 调试串口

两个 MCU 各配置一个 USART1 虚拟终端，波特率 9600 bps，8N1 格式。从机输出实时采样值，主机输出解析后的数据及报警状态，便于仿真过程中观察系统运行状态。

### 4.2 系统时钟配置

选用 HSI 8MHz 作为系统时钟源，不使用 PLL。配置代码如下：

```c
void SystemClock_Config(void) {
  RCC_OscInitTypeDef RCC_OscInitStruct = {0};
  RCC_ClkInitTypeDef RCC_ClkInitStruct = {0};

  // 开启 HSI，关闭 HSE 和 PLL
  RCC_OscInitStruct.OscillatorType = RCC_OSCILLATORTYPE_HSI;
  RCC_OscInitStruct.HSIState = RCC_HSI_ON;
  RCC_OscInitStruct.HSEState = RCC_HSE_OFF;
  RCC_OscInitStruct.PLL.PLLState = RCC_PLL_NONE;
  HAL_RCC_OscConfig(&RCC_OscInitStruct);

  // HSI 直出，AHB/APB1/APB2 均为 8MHz
  RCC_ClkInitStruct.SYSCLKSource = RCC_SYSCLKSOURCE_HSI;
  RCC_ClkInitStruct.AHBCLKDivider = RCC_SYSCLK_DIV1;
  RCC_ClkInitStruct.APB1CLKDivider = RCC_HCLK_DIV1;
  RCC_ClkInitStruct.APB2CLKDivider = RCC_HCLK_DIV1;
  HAL_RCC_ClockConfig(&RCC_ClkInitStruct, FLASH_LATENCY_0);
}
```

选择 HSI 的考虑：内部 RC 振荡器上电即振、无需外部元件、不受仿真环境晶体模型精度影响。8MHz 主频虽低于芯片标称的 72MHz，但本系统的实时性需求不高（SPI 通信 ~100ms/帧、显示更新 ~2s/次），完全满足性能要求。实物产品可换用 HSE + PLL 达到 72MHz 以获得更高吞吐量。

### 4.3 从机程序

#### 4.3.1 初始化

```
HAL_Init()                    — 系统滴答定时器
SystemClock_Config()          — HSI 8MHz
MX_GPIO_Init()                — LED、串口引脚
MX_SPI1_Init()                — SPI1 从机模式
配置 PA6 为 GPIO 推挽输出      — 软件 SPI MISO
MX_USART1_UART_Init()         — 调试串口 9600bps
MX_ADC1_Init()                — ADC1 单次转换模式
配置 PA2 为 Analog             — 温度传感器通道
LED 闪烁 → "Slave ready"
```

#### 4.3.2 主循环

从机主循环以主机轮询周期（约 500ms）为节奏运行，每次循环执行以下步骤：

1. **读气压**：ADC1 切至通道 1（PA1），单次转换并读取结果，映射为 0~600 kPa
2. **读温度**（每 4 循环一次）：ADC1 切至通道 2（PA2），读取后映射为 0~60°C
3. **组帧**：按 6 字节格式构建数据帧（同步字 0x00 + 帧头 0x5A + 气压 + 温度 + 校验和）
4. **等待片选**：轮询 PA4，等待主机拉低
5. **SPI 发送**：逐字节逐 bit 在 SCK 上升沿输出 MISO
6. **等待释放**：轮询 PA4，等待主机拉高
7. **切换 LED** + 串口调试输出

#### 4.3.3 ADC 通道管理

ADC1 配置为单次转换模式，NbrOfConversion = 1。读气压时将 SQR3 设为 CH1，读温度时设为 CH2。每次切换通道后执行完整的 Start→Poll→GetValue 周期。单次转换模式每次转换完成后 ADC 自动停止，下次启动时触发新转换，适合低速多通道轮询场景。

### 4.4 主机程序

#### 4.4.1 初始化

```
HAL_Init()
SystemClock_Config()           — HSI 8MHz
MX_GPIO_Init()                 — LED、按键、SS
MX_USART1_UART_Init(9600bps)
MX_SPI1_Init()                 — 硬件 SPI 框架
MX_TIM2_Init()                 — TIM2 CH2 PWM
SSD1306_Init()                 — OLED 初始化 + 清屏
停止 PWM + PA1 切 Analog        — 初始静音
```

#### 4.4.2 主循环

1. **SPI 接收**：拉低 SS → 延时 50ms → 重配 PA5/6/7 为 GPIO → 逐字节逐 bit 翻转 SCK 读 MISO → 拉高 SS
2. **校验解析**：校验帧头和 Checksum，通过则提取气压和温度值
3. **按键检测**：30ms 消抖，切换显示模式或报警测试标志
4. **报警判断**：气压超限或温度超限或报警测试 → alarm_state = 1
5. **蜂鸣器控制**：报警态 PA1→AF_PP→启动 PWM；正常态停 PWM→PA1→Analog（消除噪声）
6. **OLED 刷新**（每 2 秒）：概览页显示气压/温度/状态，详情页独立显示各项
7. **心跳 LED** + 串口输出 + 500ms 延时

#### 4.4.3 蜂鸣器 PWM 配置

TIM2 配置为 PWM 模式，通道 2（PA1）：
- 预分频器 PSC = 71：计数频率 = 8MHz / 72 ≈ 111.1 kHz
- 自动重载 ARR = 399：PWM 频率 = 111.1 kHz / 400 ≈ 278 Hz
- 占空比 = 50%（CCR2 = 200）

278 Hz 处于人耳最敏感频段（中频），兼顾响度和辨识度，区别于常见报警音。

### 4.5 OLED 软件 I2C 驱动

OLED 采用软件 GPIO 模拟 I2C 协议，通过直接操作 GPIO 寄存器实现 Start、Stop、SendByte 等基本操作。选择软件模拟而非硬件 I2C 外设，主要考虑：

1. **协议可控**：开发者完全掌握 I2C 时序，可自由调节通信速率
2. **简化错误处理**：省略 ACK 检查（OLED 为唯一从设备，始终响应）
3. **避免状态机冲突**：不依赖 HAL I2C 的状态机，代码逻辑清晰

I2C 引脚 PB6（SCL）和 PB7（SDA）配置为开漏输出，配合外部上拉电阻（Proteus 中 SSD1306 模型内置）。发送 1 字节时，MSB 优先，每位数据在 SCL 高电平时保持稳定，低电平时切换。

显示采用 8×16 点阵 ASCII 字库（OLED_F8x16），包含空格至 '~' 共 96 个可打印字符。每字符显示时占用 2 个 Page，全屏 4 行。

**Clear 操作优化**：清屏需向 OLED 写入 1024 字节（8 页×128 列），为 6 I2C 单次传输量最大的操作。为避免长时间占用 I2C 总线，本设计仅在初始化时执行一次 Clear，运行期间通过覆盖写方式更新显示内容，不清屏。

### 4.6 按键消抖

采用软件延时法：检测到按键按下后延时 30ms 再次确认，有效则执行对应操作并等待按键释放。30ms 延时远大于机械按键典型抖动时间（5~10ms），可有效避免误触发。

### 5.3 关键设计考量

#### 5.3.1 SPI 可靠性设计

SPI 通信的可靠性是本系统的核心。设计上采取了以下措施保障数据传输正确性：

- **同步字节**：帧首增加 0x00 字节吸收可能的位偏移
- **帧头校验**：Byte1 固定为 0x5A，主机据此判断帧对齐是否正确
- **累加和校验**：Byte5 为前 4 字节之和，主机接收后与本地计算值比对
- **低速时钟**：SCK 半周期设为 1ms，给予从机充足的 GPIO 轮询时间窗
- **失败保持**：校验失败时丢弃当前帧，保持上一个有效数据值，避免显示跳变

#### 5.3.2 ADC 采样设计

- **单次转换模式**而非连续转换：每次读取时显式触发转换，读完后 ADC 停止，避免不必要的功耗和总线占用
- **通道分时复用**：仅使用一个常规通道配置（NbrOfConversion=1），读气压/温度时动态切换 SQR3，减少 ADC 配置复杂度
- **温度低频采样**：温度变化远慢于气压，采用每 4 循环采样一次的策略，降低通道切换开销

#### 5.3.3 蜂鸣器静音设计

蜂鸣器正常态必须绝对静音，这是车载设备的基本要求。设计上通过引脚功能切换实现：报警时将 PA1 设为复用推挽输出（AF_PP），由 TIM2 PWM 驱动；正常时将 PA1 设为模拟输入（Analog），此时 PA1 的数字输出驱动器完全断开，保证在任何情况下蜂鸣器均不会意外发声。

#### 5.3.4 时钟源选择

选用内部 HSI 振荡器作为系统时钟源。HSI 作为芯片内置时钟源，上电后自动起振，无需外部晶振和匹配电容，启动速度快、可靠性高。仿真环境下 HSI 工作稳定，不受外部晶振模型精度影响。8MHz 主频满足本系统所有实时性需求。

## 六、实物系统设计建议

### 6.1 传感器选型

仿真中用电位器模拟传感器输出。实物系统推荐以下器件替换：

| 类型 | 型号 | 输出 | 量程 | 参考价 |
|------|------|------|------|--------|
| 气压 | MPX5700AP | 模拟 0.2~4.7V | 0~700 kPa | ~25 元 |
| 温度 | LM35DZ | 模拟 10mV/°C | 0~100°C | ~3 元 |
| 气压+温度一体 | MS5803-05BA | I2C/SPI | 0~500 kPa | ~35 元 |

推荐 MPX5700AP + LM35DZ 方案，均为模拟输出，直接接入 ADC，代码改动最小。MS5803 方案精度更高但需编写 I2C/SPI 驱动。

### 6.2 无线通信

实物系统需将 SPI 直连替换为无线方案。推荐 nRF24L01+ 模块，工作于 2.4GHz ISM 频段，SPI 接口控制，最高速率 2Mbps。主机和从机各配一个模块，软件流程与当前 SPI 直连一致，仅需增加 nRF24L01 的寄存器配置和收发状态机。

### 6.3 电源设计

从机（轮胎端）采用 3.7V 锂电池 + HT7333 LDO 降压至 3.3V 供电。采集周期 10 秒、每次唤醒约 50ms 的间歇工作模式下，平均电流约 0.23mA，500mAh 电池续航约 6 个月。主机（仪表台）由车载 12V 经 LM7805 和 AMS1117-3.3 两级稳压供电。

## 七、总结

本文设计并实现了一套基于 STM32F103C8T6 的直接式汽车胎压监测系统仿真原型。系统由从机采集节点和主机显示报警单元组成，通过 SPI 总线实现数据交换，完成了气压和温度的双通道 ADC 采集、SPI 数据帧封装与校验传输、SSD1306 OLED 实时显示、LED+蜂鸣器声光报警、按键交互等功能。系统采用 STM32CubeMX + HAL 库 + Keil MDK 开发，在 Proteus 9.0 平台上完成了全面的仿真验证。

在 SPI 通信、I2C OLED 驱动、ADC 采样等模块的设计中，针对仿真环境的特点选择了软件 GPIO 模拟通信协议和寄存器级外设控制方案，保证了系统的稳定运行。所有功能测试用例均通过验证，系统能够准确响应气压和温度的变化，报警逻辑正确，显示清晰。

后续可在实物平台上替换为真实传感器和无线通信模块，增加 4 轮同步监测和低功耗管理，使系统达到实用化水平。
